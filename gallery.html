📚 Media Gallery

Part D1: Component-Based Architecture with Live Search

Component Architecture: renderMediaCard()

This gallery implements the pure function component pattern taught in Week 7. Instead of hardcoding HTML repetition, a single renderMediaCard() function generates all card markup from data.

Function Signature

function renderMediaCard(item) {
  // Returns: HTML string for a single card
  // Called by: Array.map() for each media item
  // Benefits: Single source of truth, no duplication
  return `<div class="media-card">...</div>`
}

✓ Map Pattern

mediaLibrary.map(item => renderMediaCard(item)).join('') transforms data array to HTML string efficiently.

✓ Filter Pattern

mediaLibrary.filter(...) reduces dataset before mapping, enabling live search functionality.

✓ Data Separation

Data, rendering, and behaviour are fully separated. Update data without touching HTML or CSS.

Event Delegation Pattern

Rather than attaching a click listener to every card (9+ listeners), a single listener on the parent gallery listens for all clicks:

Event Flow

// Single listener on gallery container
gallery.addEventListener('click', (e) => {
  // Event bubbles up from clicked element
  const card = e.target.closest('.media-card');
  
  if (card) {
    // Handle click only if within a card
    const id = card.dataset.id;
    const title = card.dataset.title;
    // ... process click
  }
});

// Benefits:
// - 1 listener instead of 9+
// - Works for dynamically added cards
// - Lower memory footprint
// - Consistent event handling

✓ e.target.closest()

Traverses up DOM tree to find closest parent matching selector.

✓ dataset Attributes

data-id, data-type, data-title store metadata without DOM bloat.

✓ Better Performance

Event delegation scales linearly. Works identically with 10 or 10,000 cards.

✓ Responsive Grid Layout

The gallery uses CSS Grid with auto-fill and minmax() for automatic responsive columns without media queries:

grid-template-columns: repeat(auto-fill, minmax(260px, 1fr))
  • auto-fill: Creates as many columns as fit. New columns added when space available.
  • minmax(260px, 1fr): Minimum card width 260px, maximum 1 fractional unit (equal share).
  • Result: Desktop (3 cols) → Tablet (2 cols) → Mobile (1 col) automatically.

Media Library Data Structure

Each item in the media library has this structure (minimum 8 items required): 

{
  id: 'video-001',                    // unique identifier
  title: 'Learning Title',             // searchable title
  type: 'video',                       // 'video', 'audio', or 'image'
  thumbnail: 'https://...',            // image URL or data:URI
  duration: 1200,                      // seconds (null for images)
  description: 'Short description...'  // searchable text
}

Component Approach vs Hardcoded HTML

❌ Hardcoded HTML

Nine identical <div> blocks in HTML. To change card layout:

  • Edit 9 identical blocks (error-prone)
  • Changes scattered across codebase
  • Hard to maintain consistency
  • Cannot dynamically add/remove cards without JS manipulation

✓ Component Function

One renderMediaCard() function. To change card layout:

  • Edit single function (one place)
  • All cards update automatically
  • Guaranteed consistency
  • Easily scales to 50+ cards without code changes

→ Next Step: React/Vue.js Components

This component pattern is the foundation of modern frontend frameworks:

  • React: "renderMediaCard MediaCard component with props"
  • Vue: "<MediaCard v-for > with :item prop"
  • Benefit: "Same architecture, just with automatic state management and reactivity"
